① データベースの基本と関係データベース

データベースとは、データを一元管理するために集められた情報の集まりです。4大経営資源（ヒト・モノ・カネ・情報）の「情報」を管理する重要な仕組みです。

データモデル

データモデル：現実世界のデータの相互関係をわかりやすく表した図（データベースの設計図）

種類：階層データモデル、ネットワークデータモデル、関係データモデル
→ 現在最も普及しているのは関係データモデル（関係モデル）

関係データベース

関係データベースとは、データを2次元の表（テーブル）で管理するデータベースです。

用語の使い分け：

関係データベースでは…
・テーブル（表）：データの集まり
・レコード（行）：横の並び
・列：縦の並び（データの属性）

関係データモデル（設計図）では…
・関係＝テーブル、組＝レコード、属性＝列
学生テーブルの例：
学生番号学生名電話番号
1山田太郎090-9999-9999
2鈴木愛子080-8888-8888
3上峰亜衣090-1234-5678
レコード数 3、列数 3

主キーと外部キー

主キー：テーブル内のレコードを一意に識別するための列
→ 重複してはいけない / NULL（空）ではいけない
→ 例：学生番号、社員番号

複合主キー：複数の属性の組み合わせで主キーになるもの

外部キー：他のテーブルの主キーを参照する列
→ テーブル同士の関連を表す
→ 例：学生テーブルの「教員番号」が、教員テーブルの主キー「教員番号」を参照

② 正規化と関数従属

正規化とは、データベースの設計時に、データの重複や更新時の矛盾が生じないようにテーブルを整理することです。

関数従属

関数従属とは、ある属性Aの値が決まると、別の属性Bの値が一意に決まる関係です。

3つの関数従属：

完全関数従属：属性が主キー全体に関数従属している
→ 例：[学生番号] が決まると [学生名] が決まる

部分関数従属：属性が主キーの一部に関数従属している（複合主キーの場合）
→ 例：複合主キー[学生番号,試験ID]のうち、[学生番号]だけで[学生名]が決まる

推移的関数従属：属性が主キー以外の属性を介して間接的に主キーに従属
→ 例：[学生番号]→[教員番号]→[教員名]

正規化の手順

正規化のゴール：部分関数従属と推移的関数従属を取り除き、完全関数従属だけを残すこと
（＝すべての矢印が主キーから出るようにテーブルを分割）
非正規形：属性に繰り返しがある状態（正規化前）

第1正規形：属性の繰り返しがない状態にする
→ レコードを追加して繰り返しを解消

第2正規形：部分関数従属をなくす
→ テーブルを分割して、複合主キーの一部に従属する属性を別テーブルに

第3正規形：推移的関数従属をなくす
→ テーブルをさらに分割して、主キー以外の属性に従属する属性を別テーブルに
覚え方：
第1正規化 → 繰り返し除去
第2正規化 → 部分関数従属除去
第3正規化 → 推移的関数従属除去

③ SQLとデータベース管理システム（DBMS）

DBMS（DataBase Management System）とは、データベースを管理するシステムです。主な機能は「データ操作」「トランザクション管理」「排他制御」の3つです。

関係演算の3種類

選択：テーブルから条件を満たすレコード（行）を取り出す
射影：テーブルから指定した列を取り出す
結合：複数のテーブルをくっつけて1つのテーブルのように扱う

SQL ── SELECT文

SQLは、データベースを操作するための言語です。最頻出なのがSELECT文です。

SELECT文の基本構文：

SELECT 列名
FROM テーブル名
WHERE 条件式

例：学生名が「鈴木愛子」のレコードを取り出す
SELECT *
FROM 学生
WHERE 学生名 = '鈴木愛子'
SELECT文のポイント：
・SELECT 列名 → 取り出す列を指定（* ですべての列）
・FROM テーブル名 → 検索対象のテーブル
・WHERE 条件式 → 取り出すレコードの条件（省略可）
・複数列は「,」区切り
比較演算子：= 、≠（<>）、>、<、>=、<=

論理演算子：AND（かつ）、OR（または）、NOT（否定）

テーブルの結合：
FROM テーブル1, テーブル2
WHERE テーブル1.列 = テーブル2.列

ビュー

ビュー：既存のテーブルから必要な部分だけを抜き出した仮想的なテーブル
・元のテーブル（実表）のデータを変更すると、ビューにも反映される
・ビューは実データを持たず、SELECTの結果を都度生成

インデックス

インデックス（索引）：データベースの検索を高速化する機能
・「列」と「レコードの位置を指すポインタ」で構成
・本の索引と同じ仕組み
・ただし、データ更新時にインデックスの再構築が必要なため、更新は遅くなる

④ トランザクション管理・障害回復・排他制御

トランザクション

トランザクション：関連する複数の処理のまとまり（ユーザーから見た1つの処理）

例：銀行の振込処理＝「出金」＋「入金」の2処理 → 両方成功しないとダメ

ACID特性（トランザクションに必要な4つの性質）

A - 原子性（Atomicity）：トランザクション内の処理が「すべて実行される」か「すべて取り消される」かのどちらか ← 最頻出！

C - 一貫性（Consistency）：トランザクションの前後でデータベースに矛盾がない

I - 独立性（Isolation）：1つのトランザクションが他のトランザクションの影響を受けない

D - 耐久性（Durability）：完了後にハードウェア障害が発生しても結果が保証される

障害回復（リカバリ）

障害回復に必要な2つのファイル：

バックアップファイル：データベース全体を保存したファイル（定期取得）

ログファイル（ジャーナルファイル）：トランザクション前後の状態を保存
・更新前ログ：トランザクション直前の状態
・更新後ログ：トランザクション完了後の状態

ロールバック

トランザクション実行中に障害発生

→ トランザクション開始直前の状態に戻す
→ 更新前ログを使用

ロールフォワード

トランザクション完了後に障害発生

→ トランザクション完了直後の状態に戻す
→ バックアップ ＋ 更新後ログを使用

排他制御（ロック）

排他制御：複数の人が同じデータを同時に更新しようとした際、データに矛盾が生じないようにする機能

ロック：データへの読み書きを一時的に制限する機能
アンロック：ロックの解除

共有ロック

読み出しのみOK（複数可）
書き込みNG

→ データの参照時に使用

専有ロック（排他ロック）

かけた人だけ読み書きOK
他はNG

→ データの更新時に使用
第9章の最重要まとめ

・関係データベース：テーブル（表）＝レコード（行）＋列
・主キー＝レコードを一意に識別、外部キー＝他テーブルへの参照
・正規化：第1（繰り返し除去）→第2（部分関数従属除去）→第3（推移的関数従属除去）
・SELECT/FROM/WHEREの構文
・ACID：原子性・一貫性・独立性・耐久性
・ロールバック＝更新前ログ、ロールフォワード＝バックアップ＋更新後ログ
・共有ロック＝読み出しのみ、専有ロック＝排他
