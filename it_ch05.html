① 処理の仕方によるシステムの分類

システムとは、手作業をコンピュータで自動化するための仕組みです。処理の仕方には集中処理と分散処理があります。

集中処理

1台の大型コンピュータ（ホストコンピュータ）がすべてを処理

分散処理

ネットワークでつながれた複数のコンピュータが処理を分担

分散処理の3種類

クライアントサーバシステム
・クライアント（サービスを利用する側）がサーバに処理を要求
・サーバ（サービスを提供する側）が処理を実行
・クライアントに専用ソフトのインストールが必要

3層クライアントサーバシステム
・①クライアント → ②アプリケーションサーバ → ③データベースサーバ の3層
・各層が役割分担して処理する

Webシステム ← 試験最頻出！
・3層クライアントサーバシステムの一種
・クライアントソフトにWebブラウザを使う
・専用ソフトのインストール不要
・処理結果はHTML形式でクライアントに返される
Webシステムのメリット：
・使い慣れたWebブラウザが使える
・システム変更時もクライアントへのソフト再インストールが不要
・業務処理はサーバ側で実行 → クライアントは結果を表示するだけ

② システム構成と信頼性

3種類のシステム構成

シンプレックスシステム：1台だけ。最もシンプル。1箇所でも故障 → システム停止

デュプレックスシステム：現用系＋待機系の2台構成。現用系が壊れたら待機系に切替え

デュアルシステム：2台が同じ処理を同時に実行し、結果を照合。処理能力は1台分と同じ

デュプレックス

現用系が壊れたら待機系が引き継ぐ
（予備を持っておく方式）

デュアル

2台が同じ処理を行い結果を照合
（ダブルチェックの方式）

ホットスタンバイとコールドスタンバイ

デュプレックスシステムの待機系の状態には2種類あります。

ホットスタンバイ

待機系にも電源を入れて起動しておく
→ 障害時にすぐ切替え可能（高速）
→ 運用コスト高い

コールドスタンバイ

待機系は電源OFFで停止
→ 切替えに時間がかかる
→ 運用コスト安い

信頼性を高める設計

フォールトトレラント：システムが部分的に故障しても、全体として必要な機能を維持する設計

フェールソフト：障害時に性能が落ちても動作を継続させる設計（例：エンジン1つ故障でも飛行継続）
※ 限定的に動作している状態を「縮退運転（フォールバック）」という

フェールセーフ：障害時に安全を優先してシステムを停止（例：人が近づいたらロボットアーム停止）

フールプルーフ：人間が誤った操作をしてもシステムに異常が起こらない設計（例：ドアを閉めないと動かない電子レンジ） ← 頻出！

RAID

RAID（Redundant Arrays of Inexpensive Disks）は、複数のハードディスクを1つのように扱う技術です。

試験に出る3つのレベル：

RAID 0（ストライピング）：データを分散して書き込む
→ 読み書きが高速化。ただし信頼性は上がらない（むしろ下がる）

RAID 1（ミラーリング）：同じデータを複数のディスクにコピー
→ 信頼性が向上。ただし保存容量は半減、コスト増

RAID 5：データ＋パリティ（誤り訂正符号）を分散して書き込む
→ 信頼性と容量のバランスが良い

③ システムの性能指標

3つの性能指標

スループット：単位時間あたりに処理される仕事の量（「何件できるか」）

ターンアラウンドタイム：入力→処理→出力の全工程にかかる合計時間

レスポンスタイム（応答時間）：処理だけにかかる時間（入力・出力の時間は含まない）
違いの覚え方：
ターンアラウンドタイム = 入力＋処理＋出力（全部入り）
レスポンスタイム = 処理だけ（入力・出力は含まない）

ベンチマークテスト

ベンチマークテストとは、標準的なプログラムやデータを使ってシステムの処理性能を測定するテスト。

キーワード：「使用目的に合わせて選定した標準的なプログラム」

スケールアウトとスケールアップ

スケールアウト

サーバの台数を増やすことでシステム全体の処理能力を向上

スケールアップ

個々のサーバの性能を上げる（CPU交換、メモリ増設など）ことで処理能力を向上
覚え方：
スケールアウト = 外（台数）を増やす
スケールアップ = 上（性能）を上げる

④ システムの信頼性指標と稼働率の計算

RASIS（5つの評価指標）

Reliability（信頼性）── 障害の起こりにくさ
Availability（可用性）── いつでもサービスが使えるか
Serviceability（保守性）── 復旧のしやすさ
Integrity（完全性）── データが壊れていないこと
Security（安全性）── 許可された人だけがアクセスできること

MTBF と MTTR

MTBF（平均故障間隔） = 稼働時間の合計 ÷ 故障回数

→ 故障せずに動いていた時間の平均。大きいほど信頼性が高い
→ MTBFを長くするには予防保守が有効

MTTR（平均修復時間） = 故障時間の合計 ÷ 故障回数

→ 修理にかかった時間の平均。小さいほど保守性が高い
例：あるシステムが 50時間稼働→故障修理50時間→70時間稼働→故障修理70時間→60時間稼働→故障修理60時間
MTBF = (50+70+60) ÷ 3 = 60時間
MTTR = (50+70+60) ÷ 3 = 60時間

稼働率

稼働率 = MTBF ÷ (MTBF + MTTR)

→ システムが故障せずに動いていた時間の割合
→ 稼働率1.0 = 一度も故障していない
例：MTBF=3,000時間、MTTR=1,000時間
稼働率 = 3,000 ÷ (3,000+1,000) = 3,000 ÷ 4,000 = 0.75 (75%)

直列・並列システムの稼働率

直列システム（どちらも動かないとダメ）：
稼働率 = A × B

例：A=0.9, B=0.8 → 0.9 × 0.8 = 0.72

並列システム（どちらか動けばOK）：
稼働率 = 1 − (1−A) × (1−B)

例：A=0.9, B=0.8 → 1 − (1−0.9)×(1−0.8) = 1 − 0.1×0.2 = 1 − 0.02 = 0.98
稼働率の計算のコツ

直列：かけ算（両方動く確率）→ 稼働率は下がる
並列：「1 − 両方壊れる確率」→ 稼働率は上がる

多重化システム：小さい部分から順に計算 → 最後に全体を計算
MTBF改善の計算問題（頻出パターン）：
今年度：MTBF=3,000時間、MTTR=1,000時間
翌年度：MTBFを20%改善、MTTRを10%改善

新MTBF = 3,000 + 3,000×0.2 = 3,600時間（増える）
新MTTR = 1,000 − 1,000×0.1 = 900時間（減る）
稼働率 = 3,600 ÷ (3,600+900) = 3,600 ÷ 4,500 = 0.8 (80%)
